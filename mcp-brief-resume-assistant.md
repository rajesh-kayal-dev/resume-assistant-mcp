## Problem Statement
Job seekers (especially freshers and students) apply to hundreds of jobs without knowing why they get rejected. They suspect their resume isn't passing the Applicant Tracking System (ATS), but they have no way to verify how an ATS actually reads their PDF/DOCX files or how well they match a specific job description. 

## Core Tools (MCP Functions)
List 5-10 MCP tools the server should expose:

| Tool Name | Description | Input | Output |
|-----------|-------------|-------|--------|
| `parse_resume_local` | Reads a local PDF or DOCX file and extracts raw text exactly how an ATS would see it. | `file_path` (string) | `parsed_text` (string), `word_count` (int) |
| `fetch_job_description` | Scrapes the raw text of a job posting from a URL using a reading API (like Jina Reader or Firecrawl). | `url` (string) | `job_text` (string), `company_name` (string) |
| `calculate_ats_match` | Compares parsed resume text against scraped job text to calculate keyword overlap and density. | `resume_text` (string), `job_text` (string) | `match_score` (0-100), `missing_keywords` (array) |
| `generate_interview_questions` | Returns 5 highly specific interview questions based on the candidate's resume and the job requirements. | `resume_text` (string), `job_text` (string) | `questions` (array of strings) |
| `get_resume_improvements` | Analyzes resume structure (e.g., missing contact info, weak action verbs) and returns actionable tips. | `resume_text` (string) | `suggestions` (array of strings) |

## Data Strategy
- **Type**: Hybrid (Computation-heavy for local file parsing + Data-dependent for web scraping).
- **Day 1 plan**: 
  1. Use `unpdf` or `pdf-parse` for local ATS-style PDF parsing (no cost, instant).
  2. Use `r.jina.ai` (Jina Reader API) or Firecrawl free tier to scrape job descriptions from URLs (instant, free).
- **Storage**: None (stateless). All processing happens in-memory on the provided files/URLs.
- **Estimated data volume**: none — all data fetched/parsed on-demand
- **Data freshness**: Real-time API for job scraping.

> ⚠️ **Data integrity rule**: All data in this server comes from real sources. No hardcoded sample or mock databases. If scraping is needed, a data collection script must be built and run before the server can function.

### Data acquisition roadmap:
- **MVP (week 1)**: Use `unpdf` (local) and `r.jina.ai` (free web scraping) to get started immediately.
- **Growth (month 2-3)**: If Jina Reader rate limits become an issue, integrate Firecrawl API.
- **Scale (month 6+)**: Add integrations with actual ATS sandbox APIs (Greenhouse/Lever) if available.

## User Persona & JTBD (Job-to-be-Done)

### Primary Persona
- **Who**: Junior Software Engineer / CS Student
- **Context**: Applying to 10+ jobs a day via various portals.
- **Pain**: Constantly rewriting resumes, unsure if the keywords actually match what the ATS looks for.
- **Job**: "When I find an interesting job posting, I want to instantly know my ATS match score and what keywords I'm missing, so I can optimize my resume before submitting."
- **Willingness to pay**: Will pay the price of a coffee ($5) if it guarantees their resume actually gets read by a human.

### Usage Scenarios
1. **Scenario A**: User types "Check if my `resume.pdf` is good for this URL: [job link]". The server parses the PDF, scrapes the job, and tells the user exactly which bullet points to rewrite.
2. **Scenario B**: User types "Mock interview me for this role based on my resume." The server generates 5 technical questions connecting the user's specific past projects to the job's stated requirements.

| Scenario | Expected Behavior | Fallback |

| Multi-column PDF parsing | Return text as parsed, warn: "Multi-column layouts may parse incorrectly." | Recommend standard single-column ATS templates |
